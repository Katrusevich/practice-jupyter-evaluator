# AI-грейдер лабораторних робіт у форматі Jupyter Notebook

Програмний комплекс для автоматизованого оцінювання лабораторних робіт студентів, виконаних у форматі `.ipynb`. Замість класичних `assert`-тестів перевірка виконується мультимодальною AI-моделлю (Google Gemini через LangChain), яка аналізує код, текстові пояснення та графіки студента одночасно і формує структурований відгук.

## Як це працює

Файл ноутбука **не виконується повторно на сервері**. Система аналізує вже наявні в `.ipynb` дані — код комірок, Markdown-пояснення та збережені виводи (графіки у base64) — і надсилає їх мультимодальній моделі для оцінки. Це свідоме архітектурне рішення: воно прибирає ризики виконання довільного студентського коду (нескінченні цикли, переповнення пам'яті) і натомість дозволяє оцінювати змістовну частину роботи — коректність висновків, підписи на графіках тощо.

## Конвеєр (pipeline)

```
main.py
  │
  ├─ 1. parser.py         → зчитує .ipynb, групує комірки за тегами завдань
  │                          (код / текст / графіки для кожного завдання)
  │
  ├─ 2. llm_evaluator.py   → надсилає код+текст+графіки в Google Gemini,
  │                          отримує строго структуровану (Pydantic) оцінку
  │
  └─ 3. reporter.py        → формує самодостатній HTML-звіт із прогрес-барами,
                             розбивкою балів і фідбеком від AI
```

Кожен крок реалізований як незалежний модуль, дані між ними передаються простими словниками — це дозволяє замінити будь-яку частину (наприклад, формат звіту чи модель) без переписування решти логіки.

## Модулі

| Файл               | Призначення                                                                                                                                                                                                                                                                          | Статус у поточному конвеєрі                                                                                                                                        |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `main.py`          | Оркестратор: CLI, завантаження `grading_config.json`, виклик решти модулів, керована затримка `time.sleep(20)` між запитами до Gemini (захист від `429 Too Many Requests`), підсумковий вивід у консоль                                                                              | ✅ активний                                                                                                                                                         |
| `parser.py`        | Нормалізує `.ipynb` через `nbformat`, реалізує логіку «активного тегу» — угруповує текст/код/графіки за тегами завдань                                                                                                                                                               | ✅ активний                                                                                                                                                         |
| `llm_evaluator.py` | Клас `LLMEvaluator`: системний промпт з рубрикою (`rubric.md`), мультимодальний запит (текст + `image_url` з base64) до `ChatGoogleGenerativeAI`, структурований вивід через Pydantic-схему `TaskEvaluation` (`code_score`, `text_score`, `graphs_score`, `bonus_score`, `feedback`) | ✅ активний                                                                                                                                                         |
| `reporter.py`      | Генерує HTML-звіт: картки з підсумковим балом, прогрес-бари по кожному завданню, розбивка балів, текстовий AI-фідбек, галерея графіків                                                                                                                                               | ✅ активний                                                                                                                                                         |
| `executor.py`      | `NotebookExecutor` — ізольований запуск коду через `nbclient`/`KernelManager` з таймаутом і `allow_errors=True`                                                                                                                                                                      | ⚠️ **у коді є, але `main.py` його не імпортує й не викликає.** Залишений як напрацювання з ранньої стадії проєкту, коли планувалося реально виконувати код студента |
| `grader.py`        | `Grader` — класичне оцінювання на основі проходження `assert`-тестів у тестових комірках                                                                                                                                                                                             | ⚠️ **не використовується.** Це підхід попередньої (до-AI) версії системи                                                                                            |

> **Чому `executor.py` і `grader.py` лишились у репозиторії:** це артефакти першого варіанту архітектури (виконання коду → перевірка assert'ами), від якого свідомо відмовились на користь AI-аналізу. Файли не видалені, оскільки документують хід розробки й можуть знадобитися, якщо в майбутньому знадобиться повернути гібридний режим (assert + AI).

## Система тегів

Теги завдань (`task-1-data`, `task-2-ann`, `task-3-keras`, `task-4-conclusions` — див. `grading_config.json`) **проставляються вручну студентом** у метаданих комірки Jupyter (`cell.metadata.tags`), а не генеруються автоматично.

`parser.py` не вимагає тегувати кожну комірку окремо: досить позначити тегом **першу** комірку завдання — далі парсер запам'ятовує цей «активний тег» і автоматично приєднує до нього весь наступний код і текст, аж поки не зустріне тег наступного завдання. Це й реалізує «логіку активного тегу», описану в основній частині звіту.

⚠️ Зверніть увагу: `sample_student.ipynb` у репозиторії використовує **інші** теги (`solution`, `autograder-test`) — вони належать до старої `assert`-логіки (`grader.py`) і **не розпізнаються** поточним `parser.py`/`grading_config.json`. Для тестування поточного AI-конвеєра потрібен ноутбук із тегами `task-1-data` / `task-2-ann` / `task-3-keras` / `task-4-conclusions`.

## Конфігурація

### `grading_config.json`
Визначає структуру оцінювання без втручання у Python-код:
- `required_files` — датасети, які має використовувати ноутбук;
- `parts` — перелік завдань: `id`, назва (`name`), тег для пошуку в ноутбуці (`tag`), максимальний бал (`max_points`).

### `rubric.md`
Текстова рубрика критеріїв, яка підвантажується в системний промпт `LLMEvaluator`. Бали в рубриці (15 / 25 / 25 / 15, разом 80 + 20 бонусних) узгоджені з `max_points` у `grading_config.json`.

## Встановлення

```bash
pip install -r requirements.txt
pip install langchain-google-genai langchain-core pydantic python-dotenv
```

> ⚠️ **`requirements.txt` неповний.** Він містить лише пакети для роботи з ноутбуками (`nbformat`, `nbclient`, `nbconvert`, `jupyter_client`, `ipykernel`) — це залежності старого, assert-based підходу. Пакети, без яких **не запуститься сама AI-частина** (`langchain-google-genai`, `langchain-core`, `pydantic`, `python-dotenv`), у файлі відсутні. Це варто виправити, доповнивши `requirements.txt` явно.

Створіть файл `.env` у корені проєкту:
```
GOOGLE_API_KEY=ваш_ключ_google_ai_studio
```

## Використання

```bash
python main.py шлях/до/ноутбука.ipynb
python main.py шлях/до/ноутбука.ipynb -v   # детальне (DEBUG) логування
```

Результат — файл `report_<ім'я_ноутбука>.html` у поточній директорії та стислий підсумок балів у консолі.

`test.py` — допоміжний скрипт для перевірки, які моделі Gemini доступні за вашим API-ключем (`genai.list_models()`); до основного конвеєра не належить і використовує окрему бібліотеку (`google.generativeai`) замість `langchain_google_genai`.

## Приклад згенерованого звіту

`report_Kasianova_Kate_lab_1.html` — звіт, згенерований поточним `reporter.py` (картки балів, прогрес-бари, AI-фідбек, галерея графіків).

`report_sample_student.html` — звіт **старого** формату (таблиці критеріїв, окремий блок `Assert-тести`, попередження про відсутні датасети). Він згенерований попередньою версією `reporter.py`, яка вже не відповідає коду, що зараз лежить у репозиторії.

## Відомі обмеження

- Немає повторних спроб (`retry`) запитів до Gemini — лише статична затримка `sleep(20)` в `main.py`.
- `executor.py` та `grader.py` не підключені до конвеєра — потребують явного рішення: видалити чи повернути в дію.
- `requirements.txt` не відображає реальний стек проєкту.
- Контейнеризація (Docker) для ізоляції — у планах, у коді відсутня.
